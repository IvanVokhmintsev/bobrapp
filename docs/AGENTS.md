# AGENTS.md — правила для AI-агента

Правила работы AI-агента (Cursor / Codex) в проекте **Music Roadmap App** (bobrapp).

---

## 1. Источники истины (приоритет)

При конфликте следуй порядку:

1. **`docs/REQUIREMENTS.md`** — реестр требований из отчёта (главный источник).
2. **`docs/PRODUCT_SPEC.md`** — продуктовая спецификация MVP.
3. **UI / Figma / `figma_ref/`** — ориентир по вёрстке, если не противоречит требованиям.
4. **Текущий код** — сохраняй существующие паттерны.

> Если макет и отчёт расходятся — **приоритет у отчёта** (`REQUIREMENTS.md`).  
> Если в отчёте нет детали — делай минимально разумную реализацию в стиле приложения, не придумывай новые фичи.

Перед началом задачи прочитай релевантные разделы:

- `docs/REQUIREMENTS.md`
- `docs/README.md`
- `docs/PRODUCT_SPEC.md`
- этот файл — `docs/AGENTS.md`

Файл `docs/TASKS.md` **устарел** — не используй его как чеклист готовности; ориентируйся на `REQUIREMENTS.md` и код.

---

## 2. Автономия и общение

**По умолчанию работай самостоятельно:**

- не жди подтверждения на каждый шаг;
- разбивай задачу на подзадачи и выполняй итерациями;
- чини сопутствующие баги, если они блокируют сценарий из требований.

**Спроси пользователя только если:**

- в `REQUIREMENTS.md` противоречие или неоднозначность;
- нужно продуктовое решение, которое нельзя вывести из отчёта;
- задача требует destructive-действий (сброс БД, force push, удаление данных);
- пользователь явно попросил «сначала план / без изменений файлов».

**План:**

- для мелких фиксов — сразу делай;
- для крупных фич (несколько экранов, schema + API + UI) — краткий план в 3–7 пунктов, затем реализация без паузы, если пользователь не просил иначе.

**Git:**

- **коммить** после каждой логически завершённой итерации (собирается, имеет смысл отдельно);
- сообщение коммита — кратко **зачем**, на английском или русском, в стиле репозитория;
- **не push** без явной просьбы;
- не менять git config, не делать force push, не коммить `.env` и секреты;
- не коммить, если build падает (сначала почини).

Примеры границ коммита: «fix logout redirect», «feat(profile): solo/band profile type», «docs: update AGENTS.md».

## 3. Структура репозитория

```
backend/          — API: Fastify + TypeScript + Prisma + PostgreSQL
frontend/web/     — Web UI: React + Vite + React Router
docs/             — требования, спецификация, правила агента
```

Package manager: **pnpm** (корень монорепо и/или отдельные `package.json` в `backend`, `frontend/web`).

---

## 4. Технический стек (фактический)

### Backend (`backend/`)

- Fastify, TypeScript
- Prisma ORM, PostgreSQL
- Auth: **httpOnly signed cookie** `bobrapp_session` (не JWT в localStorage)
- Загрузки: `backend/uploads/` (аватары, медиа постов)
- Модули: `src/modules/<domain>/` (routes, schemas, types, presenter/service)

### Frontend (`frontend/web/`)

- React 19, Vite, React Router
- API client: `src/api.ts` (`credentials: "include"`)
- Auth state: `src/context/AuthContext.tsx`
- Layout: sidebar + mobile tab bar (`AppSidebar`, `AppTabBar`)
- Стили: feature-level CSS, общий shell — `app-shell.css`

### Env

- Секреты только в `.env` (не коммитить).
- Примеры — `.env.example`.
- Локально: `VITE_API_URL=http://localhost:3000`, backend `:3000`, frontend `:5173`.

## 5. Что уже реализовано (не ломай без причины)

- Регистрация / вход / logout (cookie session)
- Онбординг музыканта (уровень + тип профиля solo/band)
- Лента: посты, лайки, комментарии, фото/аудио
- Roadmap: этапы, урок, quiz, прогресс
- Профиль: свой и публичный `/profile/:userId`
- Музыканты (`/people`), заглушки Букинг / События
- Подписки (follow), достижения, базовое редактирование профиля

При доработке **расширяй**, а не переписывай архитектуру целиком.

---

## 6. Стиль разработки

- **Минимальный diff** — только то, что нужно для задачи.
- **Маленькие итерации** — одна логически завершённая цель за проход.
- Backend-задача → не трогай frontend без необходимости; и наоборот.
- **Не добавляй библиотеки** без веской причины; сначала используй то, что уже в проекте.
- Код простой и читаемый; комментарии — только для неочевидной логики.
- Новые API: валидация (schemas), auth middleware, понятные ошибки, presenter для ответов.
- **Prisma:** только новые миграции; не редактируй уже применённые migration.sql.
- **Приватность:** пользователь не должен видеть чужие приватные данные.

---

## 7. Вне scope (без явного запроса)

Не реализовывай, если этого нет в задаче / `REQUIREMENTS.md`:

- оплату, подписки, trial;
- продвижение постов/профилей;
- матчинг музыкант–лейбл, сложные рекомендации;
- чаты, push-уведомления;
- аналитику и сложную модерацию;
- хранение больших аудио/видео сверх текущей модели постов.

---

## 8. UI и дизайн

- Figma / макеты — **ориентир**, не догма.
- Если блок не проработан в дизайне — используй существующие паттерны: `app-page`, `app-page__panel`, sidebar, tab bar.
- **Функциональность из отчёта важнее пиксель-перфекта.**
- Кнопки и сценарии должны **работать** (навигация, API, auth).
- **Тёмная тема:** не используй голый HTML-тег `<header>` в экранах приложения — глобальные legacy-стили ломают фон. Заголовки секций и модалок — `<div className="…__header">` с явным `background` и `color` в feature-CSS.

---

## 9. Проверки перед завершением задачи

Запускай сам, не проси пользователя «нажать кнопку», если можешь проверить локально.

| Область | Команды |
|--------|---------|
| Backend | `cd backend && pnpm install` (если нужно), `pnpm exec prisma migrate deploy` (если менялась schema), `pnpm run build` |
| Frontend | `cd frontend/web && pnpm run build` |

- **`pnpm dev`** — для локального просмотра, не обязательный шаг верификации после каждой задачи.
- Если тестов нет — не добавляй их без запроса; достаточно сборки и ручного сценария проверки.

---

## 10. Формат отчёта после задачи

Кратко, по делу:

1. **Что сделано** (при возможности — ID из `REQUIREMENTS.md`, напр. `FR-PP-01.2`).
2. **Какие файлы изменены.**
3. **Как проверить** (3–5 шагов руками).
4. **Что осталось** / известные ограничения.

Не раздувай ответ; diff в IDE и так виден.

---

## 11. Типовой workflow

1. Прочитать задачу и связанные пункты `docs/REQUIREMENTS.md`.
2. Быстро оценить затронутые backend / frontend / schema.
3. Реализовать минимальный рабочий vertical slice.
4. Прогнать build / migrate.
5. Исправить ошибки.
6. Дать краткий отчёт (§10).

---

## 12. Делегирование от владельца проекта

Пользователь может дать задачу в формате:

```text
Закрой [описание или ID из REQUIREMENTS.md].
Приоритет: отчёт. UI — по Figma где есть.
Работай автономно. Коммить по итерациям. Push — только если попрошу.
```

Этого достаточно — **не требуется** пошаговое ТЗ на каждое поле и endpoint.
