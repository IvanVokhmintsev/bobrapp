# AGENTS.md — Instructions for Codex

Этот файл задает правила работы для Codex / AI-агента в проекте Music Roadmap App.

## General Rules

1. Перед изменением кода сначала кратко описывай план.
2. Не меняй архитектуру без согласования.
3. Не переписывай весь проект ради одной маленькой задачи.
4. Не добавляй новые библиотеки без объяснения причины.
5. После изменений показывай список измененных файлов.
6. Секреты, токены и ключи храни только в `.env`.
7. Для примеров используй `.env.example`, а не реальные ключи.
8. Если задача неясна, сначала задай уточняющий вопрос или предложи безопасное предположение.
9. Код должен быть простым, читаемым и понятным для начинающего разработчика.
10. После реализации backend запускай проверки из папки `backend`: `pnpm install`, `pnpm dev`, `pnpm test`, если тесты есть.

## Project Context

Перед началом работы прочитай:

- `README.md`
- `PRODUCT_SPEC.md`
- `TASKS.md`

## MVP Boundaries

В рамках первой версии MVP реализуются:

- регистрация и вход;
- роли `musician` и `label`;
- онбординг музыканта;
- социальная лента;
- roadmap;
- урок roadmap;
- прогресс пользователя;
- профиль;
- достижения.

## Do Not Implement Without Explicit Request

Не реализовывай без отдельного запроса:

- оплату;
- подписку;
- trial period;
- продвижение постов;
- продвижение профилей;
- расширенный поиск лейблов;
- матчинг музыкантов и лейблов;
- сложные рекомендации;
- чаты;
- аналитику;
- загрузку и хранение больших аудио/видео-файлов;
- сложную модерацию;
- push-уведомления.

## Development Style

- Делай изменения маленькими итерациями.
- Один запрос — одна логически завершенная задача.
- Backend пишется на TypeScript с Fastify, если отдельно не согласовано другое.
- Если задача касается backend, не трогай frontend без необходимости.
- Если задача касается frontend, не меняй backend без необходимости.
- Для новых endpoints добавляй базовую обработку ошибок.
- Для защищенных endpoints учитывай текущего пользователя.
- Не давай пользователю видеть чужие приватные данные.

## Response Format After Each Task

После выполнения задачи напиши:

1. Что было сделано.
2. Какие файлы изменены.
3. Какие команды нужно запустить.
4. Как проверить результат.
5. Какие ограничения или TODO остались.

## Preferred Task Flow

1. Прочитай задачу.
2. Сформулируй короткий план.
3. Дождись подтверждения, если пользователь попросил не менять файлы без согласования.
4. Внеси изменения.
5. Запусти проверки.
6. Исправь ошибки, если они появились.
7. Дай краткий отчет.
